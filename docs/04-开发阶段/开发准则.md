# Vexfy 开发准则

## 一、技术栈

### 1. 状态管理 - Riverpod

Vexfy 使用 `flutter_riverpod` 作为状态管理方案，替代原来的 GetX。

**核心概念：**
- `Notifier` / `NotifierProvider` - 用于管理可变更的状态
- `Provider` - 用于提供不可变的数据
- `ConsumerWidget` / `ConsumerStatefulWidget` - 用于在 Widget 中访问 Riverpod 状态

**示例：**
```dart
// 定义状态类
class PlayerState {
  final List<SongModel> playlist;
  final SongModel? currentSong;
  final bool isPlaying;
  final String? error; // 错误信息字段
  // ...
}

// 定义 Notifier
class PlayerNotifier extends Notifier<PlayerState> {
  @override
  PlayerState build() {
    _initPlayer();
    return const PlayerState();
  }

  void play() { /* ... */ }
  void pause() { /* ... */ }
}

// 定义 Provider
final playerProvider = NotifierProvider<PlayerNotifier, PlayerState>(() {
  return PlayerNotifier();
});

// 在 Widget 中使用
class MyWidget extends ConsumerWidget {
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final playerState = ref.watch(playerProvider);
    final notifier = ref.read(playerProvider.notifier);

    return Text('Current: ${playerState.currentSong?.title}');
  }
}
```

**规范：**
1. 每个业务模块对应一个 `Notifier`（如 `PlayerNotifier`、`LocalMusicNotifier`）
2. 状态类使用 `copyWith` 方法支持部分更新
3. Widget 使用 `ConsumerWidget` 而非普通 `StatelessWidget`
4. 使用 `ref.watch()` 监听状态变化，使用 `ref.read()` 执行操作
5. **状态类必须有 `error` 字段**，用于存储错误信息供 UI 显示

### 2. 路由管理 - go_router

Vexfy 使用 `go_router` 作为路由管理方案，替代原来的 GetX 路由。

**核心概念：**
- `GoRouter` - 路由配置
- `GoRoute` - 单个路由定义
- `context.go()` / `context.push()` - 路由跳转

**示例：**
```dart
// router/app_router.dart
class AppRouter {
  static final GoRouter router = GoRouter(
    initialLocation: AppRoutes.home,
    routes: [
      GoRoute(
        path: AppRoutes.home,
        name: 'home',
        builder: (context, state) => const HomePage(),
      ),
      GoRoute(
        path: AppRoutes.player,
        name: 'player',
        pageBuilder: (context, state) => CustomTransitionPage(
          child: const PlayerPage(),
          transitionsBuilder: (context, animation, secondaryAnimation, child) {
            return SlideTransition(
              position: Tween<Offset>(
                begin: const Offset(0, 1),
                end: Offset.zero,
              ).animate(animation),
              child: child,
            );
          },
        ),
      ),
    ],
  );
}

// 路由名称常量
abstract class AppRoutes {
  static const String home = '/';
  static const String player = '/player';
}

// 跳转示例
context.go(AppRoutes.home);      // 替换当前页
context.push(AppRoutes.player);  // 入栈新页
```

**规范：**
1. 所有路由配置集中在 `router/app_router.dart`
2. 路由名称定义在 `AppRoutes` 抽象类中
3. 跳转使用 `context.go()`（不需要返回）或 `context.push()`（需要返回）
4. 页面使用 `ConsumerWidget` 以支持 Riverpod

### 3. 日志系统 - talker

Vexfy 使用 `talker` 作为日志系统，替代原来的 `logging` + `print`。

**核心概念：**
- `Talker` - 全局日志实例
- `talker.info()` / `talker.debug()` / `talker.severe()` - 日志级别
- `TalkerWrapper` - Flutter 应用日志监控

**示例：**
```dart
// main.dart
final talker = Talker();

void main() async {
  talker.info('[main] App 启动');
  talker.debug('[main] Debug 信息');
  talker.severe('[main] 错误', exception: e, stackTrace: s);

  runApp(
    ProviderScope(
      child: MaterialApp.router(
        builder: (context, child) {
          return TalkerWrapper(talker: talker, child: child);
        },
      ),
    ),
  );
}
```

**规范：**
1. 全局 `talker` 实例在 `main.dart` 顶部定义
2. 统一使用 `talker.info()` / `talker.debug()` / `talker.severe()` 输出日志
3. 异常信息使用 `exception` 和 `stackTrace` 参数
4. 生产环境自动关闭 debug 日志

### 4. 音频播放 - just_audio + just_audio_mpv

Vexfy 使用 `just_audio` 作为音频播放核心，`just_audio_mpv` 提供 Linux 桌面支持。

**核心概念：**
- `AudioPlayer` - just_audio 播放器实例
- `playerProvider` - Riverpod 管理的播放器状态
- `audio_service` - 后台播放 + 通知栏控制

**Linux 桌面依赖：**
```bash
# Ubuntu/Debian
sudo apt install libmpv-dev

# Fedora
sudo dnf install mpv-devel

# Arch Linux
sudo pacman -S libmpv
```

**规范：**
1. 播放器逻辑封装在 `providers/player_provider.dart` 的 `PlayerNotifier` 中
2. 后台服务使用 `audio_service`，通过 `services/audio_handler_service.dart` 管理
3. 播放状态通过 Riverpod 暴露给 UI 层

### 5. 权限管理 - permission_handler

Vexfy 使用 `permission_handler` 处理存储权限，替代原来的 `on_audio_query_pluse`。

**核心概念：**
- `Permission.storage` - Android 存储权限
- `Permission.audio` - Android 13+ 音频权限
- `file_picker` - 用户选择音乐目录

**示例：**
```dart
// 请求存储权限
final status = await Permission.storage.request();
if (status.isGranted) {
  // 有权限
}

// 让用户选择音乐目录
final dir = await FilePicker.platform.getDirectoryPath(
  dialogTitle: '选择音乐目录',
);
```

**规范：**
1. 权限请求封装在 `LocalMusicNotifier` 中
2. 用户拒绝权限时，降级到让用户手动选择目录
3. 使用 `file_picker` 让用户自定义扫描目录

## 二、日志打印规范（核心原则）

> **所有关键步骤都要有日志，关键逻辑要打印入参出参**

### 日志级别使用规范

| 级别 | 使用场景 |
|------|----------|
| `talker.info()` | 关键流程开始/结束，如 App 启动、扫描完成、播放成功 |
| `talker.debug()` | 详细调试信息，如每次操作、状态变化 |
| `talker.warning()` | 异常但可恢复，如权限被拒绝、文件不存在 |
| `talker.severe()` | 严重错误，如初始化失败、崩溃 |

### 日志格式规范

```dart
// ✅ 正确示例
talker.info('[播放] 开始播放: ${song.title}');
talker.debug('[控制] togglePlayPause()');
talker.warning('[权限] 权限被拒绝，降级到手动选择');
talker.severe('[初始化] 播放器初始化失败', exception: e, stackTrace: s);

// ❌ 错误示例
print('playing');                    // 无上下文
log('error: $e');                   // 无日志级别
```

### 日志打印位置（强制）

1. **每个 async 函数入口** - 打印入参
2. **每个 async 函数出口** - 打印出参或执行结果
3. **关键业务逻辑** - 打印入参、出参、中间状态
4. **异常捕获** - 打印异常和堆栈

```dart
// ✅ 标准模板
Future<void> playSong(SongModel song) async {
  talker.info('[播放] playSong() 入参: song=${song.title}');

  try {
    // 业务逻辑
    talker.fine('[播放] 加载文件: $filePath');
    await _player.setFilePath(filePath);
    talker.info('[播放] 播放成功: ${song.title}');
  } catch (e, s) {
    talker.severe('[播放] playSong 异常', exception: e, stackTrace: s);
    state = state.copyWith(error: '播放失败: $e');
  }
}
```

## 三、异常处理规范（核心原则）

> **每个 async 函数都要有 try-catch，不能让异常裸奔**

### 强制要求

1. **每个 async 函数必须包裹 try-catch**
2. **catch 块必须记录异常**（使用 talker）
3. **catch 块必须设置 error 状态**（让 UI 层知道出错了）
4. **catch 块不能 rethrow（除非是顶层需要处理）**

```dart
// ✅ 正确示例
Future<void> playSong(SongModel song) async {
  talker.info('[播放] playSong() 入参: song=${song.title}');

  try {
    // 业务逻辑
    await _switchToIndex(idx);
    await _player.play();
    talker.info('[播放] 播放成功');
  } catch (e, s) {
    talker.severe('[播放] playSong 异常', exception: e, stackTrace: s);
    state = state.copyWith(error: '播放失败: ${e.toString()}');
    // 不 rethrow，让 App 继续运行
  }
}

// ❌ 错误示例
Future<void> playSong(SongModel song) async {
  // 没有 try-catch，异常会向上传播
  await _player.play();
}
```

### 异常类型处理

| 异常类型 | 处理方式 |
|----------|----------|
| 播放器初始化失败 | 设置 `error = '播放器暂不可用'`，让 UI 显示 |
| 数据库读写失败 | 降级到内存模式，或设置 `error` |
| 文件不存在 | 设置 `error = '文件不存在或已删除'` |
| 权限被拒绝 | 提示用户手动选择，不崩溃 |
| 网络请求失败 | 设置 `error`，显示重试按钮 |

## 四、异常兜底规范（核心原则）

> **即使某个功能失败，App 也不能崩**

### 必须有兜底的场景

| 场景 | 兜底方案 |
|------|----------|
| 播放器初始化失败 | `error = '播放器暂不可用'`，UI 显示提示 |
| 数据库初始化失败 | 降级到内存模式 (`inMemoryDatabasePath`) |
| 音频服务启动失败 | 继续运行，播放器功能显示"暂不可用" |
| 文件加载失败 | 显示"文件不存在或已删除" |
| 权限获取失败 | 提示用户选择目录，不崩溃 |

### 启动时的异常兜底

```dart
Future<void> main() async {
  try {
    // 初始化数据库（失败降级到内存模式）
    await _initDatabaseWithFallback();

    // 初始化音频服务（失败不影响主功能）
    try {
      await startAudioService();
    } catch (e) {
      talker.warning('[main] 音频服务启动失败（继续运行）', exception: e);
    }

    runApp(const ProviderScope(child: VexfyApp()));
  } catch (e, s) {
    // 显示错误界面，不闪退
    runApp(_ErrorApp(message: '启动异常: $e'));
  }
}
```

### 播放器兜底示例

```dart
Future<void> _initPlayer() async {
  try {
    _player = AudioPlayer();
    // 初始化监听器...
    _isInitialized = true;
    talker.info('[初始化] 播放器初始化成功');
  } catch (e, s) {
    talker.severe('[初始化] 播放器初始化失败', exception: e, stackTrace: s);
    state = state.copyWith(error: '播放器初始化失败，请重启 App');
    // 不 rethrow，让 App 能继续启动
  }
}

Future<void> playSong(SongModel song) async {
  if (!_isInitialized) {
    talker.warning('[播放] 播放器未初始化');
    state = state.copyWith(error: '播放器暂不可用');
    return; // 安全返回，不崩溃
  }
  // 正常播放逻辑...
}
```

## 五、架构规范

### 1. 目录结构

```
lib/
├── main.dart                    # 入口文件（含全局异常兜底）
├── router/
│   └── app_router.dart          # go_router 路由配置
├── providers/
│   ├── player_provider.dart     # 播放器状态（含完善日志+异常处理）
│   ├── local_music_provider.dart # 本地音乐状态（含完善日志+异常处理）
│   └── tab_index_provider.dart   # Tab 索引状态
├── services/
│   └── audio_handler_service.dart # 后台播放服务（含完善日志）
├── app/
│   ├── core/
│   │   └── theme.dart           # 主题配置
│   └── modules/
│       ├── home/                # 首页
│       ├── player/              # 播放器相关
│       ├── playlist/           # 播放列表
│       ├── stats/              # 统计
│       └── settings/           # 设置
├── data/
│   ├── models/                  # 数据模型
│   └── database/               # 数据库相关（含内存兜底）
└── widgets/                     # 公共组件
```

### 2. 状态管理模式

- 每个业务模块有一个 `Notifier` + `State` 类
- `State` 类使用 `copyWith` 支持部分更新，包含 `error` 字段
- `Notifier` 管理所有业务逻辑，所有 async 方法有 try-catch
- UI 层通过 `ref.watch()` 监听状态，`ref.read()` 执行操作

### 3. 异常处理流程

```
异常发生
   ↓
catch 捕获
   ↓
记录日志 (talker.severe/warning)
   ↓
设置 error 状态
   ↓
安全返回（不 rethrow，除非顶层需要）
   ↓
UI 层通过 error 字段显示提示
   ↓
App 继续运行
```

## 九、健康检测机制（核心模块）

### 概述

健康检测服务（`HealthCheckService`）在 App 启动时全面检测各项功能是否可用，是架构的核心保障模块。

### 检测项目（12 项）

| 功能 | 检测方式 | 日志示例 |
|------|---------|---------|
| 数据库 | 写入/读取 app_info 表 | `[健康检测-数据库] ✅ 正常` |
| 文件读写 | 读写一个测试文件 | `[健康检测-文件读写] ✅ 正常` |
| 存储权限 | 检查 audio/storage 权限状态 | `[健康检测-存储权限] ✅ 已授权` |
| 网络状态 | 检测网络连通性 | `[健康检测-网络状态] ✅ 在线（wifi）` |
| 通知栏权限 | 检查 notification 权限 | `[健康检测-通知栏权限] ✅ 已授权` |
| 后台播放权限 | 检查后台音频权限 | `[健康检测-后台播放] ✅ 正常` |
| 音频焦点 | 测试 AudioPlayer 可用性 | `[健康检测-音频焦点] ✅ 正常` |
| 音频播放 | 播放 500ms 测试音（1%音量） | `[健康检测-音频播放] ✅ 正常` |
| 耳机检测 | 检测耳机插拔状态 | `[健康检测-耳机检测] ✅ 正常（运行时检测）` |
| 磁盘空间 | df -h 检查剩余空间 | `[健康检测-磁盘空间] ✅ 剩余 2.3GB` |
| OSS连接 | ping endpoint 或网络检测 | `[健康检测-OSS连接] ✅ 正常（网络可用）` |
| 内存状态 | free -h / vm_stat 检查剩余内存 | `[健康检测-内存状态] ✅ 剩余 1.2GB` |

### 核心原则

1. **每个检测都要 try-catch**，失败不影响 App 启动
2. **检测结果全部打印日志**，精确到哪个功能有问题
3. **日志格式统一**：`[健康检测] 功能名: 状态 + 详情`
4. **任何功能检测失败，要有友好提示而不是闪退**

### 使用方式

```dart
// main.dart
Future<void> main() async {
  // ...
  
  // 执行健康检测
  try {
    final allHealthy = await HealthCheckService.instance.runAllChecks();
    if (allHealthy) {
      talker.info('[main] 健康检测：全部通过');
    } else {
      final failed = HealthCheckService.instance.getFailedFeatures();
      talker.warning('[main] 健康检测：${failed.join("、")} 功能异常（App 继续运行）');
    }
  } catch (e, s) {
    talker.warning('[main] 健康检测执行异常（继续运行）', exception: e, stackTrace: s);
  }
  
  // 打印健康检测摘要
  _printHealthCheckSummary();
}

void _printHealthCheckSummary() {
  final results = HealthCheckService.instance.results;
  for (final r in results) {
    if (r.isHealthy) {
      talker.info('[main]   ✅ ${r.feature}: ${r.message}');
    } else {
      talker.warning('[main]   ❌ ${r.feature}: ${r.message}');
    }
  }
}
```

### 音频播放检测说明

音频播放检测会在启动时播放约 500ms 的测试音频（音量设为 1%），用于验证：
- just_audio 是否正常工作
- Linux 平台 libmpv 是否可用
- 音频后端是否正常

检测失败会在日志中显示友好提示，App 继续运行。

**技术经验要积累不要遗忘：**
- 每次发现新问题/新方案，都要记录到知识库
- 健康检测机制的理念可以复用
- 异常处理模式可以复用
- 打包踩坑经验要记录

**知识库维护节奏：**
- 每次迭代结束：回顾本次踩坑，更新知识库
- 每周五：团队经验分享（15 分钟）
- 每次发版：检查知识库是否同步更新
