### 2.2 ⚠️ 建议改进项

#### 2.2.1 `onUpgrade()` 为空

**问题**：`onUpgrade` 方法存在但没有任何迁移逻辑，未来版本升级时可能丢失数据。

```dart
Future<void> _onUpgrade(
    Database db, int oldVersion, int newVersion) async {
  _logger.info('[升级] 数据库升级: $oldVersion -> $newVersion');
  // 未来迁移脚本写在这里
}
```

**建议**：保持注释说明，但确保未来版本有迁移脚本。

#### 2.2.2 `songs` 表缺少 artist/album 索引

**问题**：搜索功能按 `title` 和 `artist` 搜索，但只对 `title` 建立了索引。

```dart
await db.execute(
    'CREATE INDEX idx_songs_title ON ${Tables.songs}(title)');
// 没有 idx_songs_artist
```

**建议**：如果搜索功能频繁使用 artist，应该添加 `idx_songs_artist` 索引。

#### 2.2.3 `song_stats` 表缺少播放时段字段

**问题**：PRD 要求支持"按播放时段统计（morning/afternoon/evening/night）"，但 `song_stats` 表没有对应字段。

```dart
await db.execute('''
  CREATE TABLE ${Tables.songStats} (
    song_id     TEXT PRIMARY KEY,
    play_count  INTEGER NOT NULL DEFAULT 0,
    total_seconds INTEGER NOT NULL DEFAULT 0,
    last_played_at TEXT
  )
  // 缺少: favorite_time_slot (morning/afternoon/evening/night)
  // 缺少: favorite_segment_start (30s 粒度)
)
```

**建议**：添加 `favorite_time_slot TEXT` 和 `favorite_segment_start INTEGER` 字段。

#### 2.2.4 `local_file_metadata` 表缺少唯一约束

**问题**：`local_file_metadata` 表使用 `song_id` 作为主键，但 `song_id` 来自文件路径的 MD5。如果两个文件 MD5 碰撞（极低概率），会出问题。

**建议**：可以接受，但可以考虑使用文件路径作为唯一约束。

#### 2.2.5 `SongModel.fromMap()` 的 `play_count` 默认0

**问题**：当 `play_count` 字段为 null 时，默认值是0。这是合理的，但可能丢失真实的 null 语义。

```dart
playCount: map['play_count'] as int? ?? 0,
```

**建议**：可以接受，但如果需要区分"从未播放过"和"播放了0次"，需要额外字段。
