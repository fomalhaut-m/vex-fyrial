### 对话 1：项目启动

**Luke**：Flutter 音乐APP 完整技术方案（之前给你的那套，原样整理）

**技术方案要点**：
- 采用 Flutter 分层架构（表现层/状态层/业务层/数据层）
- 状态管理：GetX
- 本地数据库：sqflite + path_provider
- 网络请求：dio
- 播放器核心：just_audio
- 后台播放：audio_service + just_audio
- 本地音乐扫描：on_audio_query
- 缓存管理：flutter_cache_manager
- 路由管理：GetX 路由

**核心功能**：
1. 闪屏页 + 引导页
2. 首页推荐、排行榜、歌单广场
3. 在线音乐播放
4. 本地音乐扫描 & 本地曲目播放
5. 歌词滚动同步（LRC 解析）
6. 底部常驻播放栏、迷你播放器
7. 播放模式：列表循环/单曲循环/随机
8. 收藏歌单、创建自定义歌单
9. 后台播放、锁屏通知栏控制
10. 音质选择、倍速播放
11. 搜索歌曲、歌手、专辑

**关键难点**：
- 后台保活+锁屏控制 → audio_service + just_audio
- 本地音乐全盘扫描 → on_audio_query
- 歌词LRC滚动 → 解析时间戳+ListView居中滚动
- 全局底部播放栏 → GetX全局状态
- 网络音乐缓存 → 已听过二次免流量

**开发顺序**：
1. 搭建项目骨架 + GetX 路由
2. 集成播放器基础播放/暂停/切歌
3. 做底部常驻播放栏
4. 接入本地音乐扫描
5. 在线API 对接推荐、排行榜
6. 歌词解析 + 滚动
7. 后台播放+通知栏
8. 歌单收藏、本地缓存、设置页

### 对话 11：音乐来源纠正（重要）

**Luke**：发现音乐就只有在本地去扫描，没有在线分析，只有像OSS同步，OSS和本地是双向同步

**纠正要点**：
- 音乐来源 = 本地扫描 + OSS双向同步，**不是**在线推荐/排行榜
- 没有在线音乐 API（不是网易云音乐模式）
- 用户有自己的 OSS 存储空间，音乐文件在本地和 OSS 之间双向同步
- App 只做本地扫描播放，不做在线内容发现
- 如果需要新增音乐，用户自己上传到 OSS 或本地，App 同步后发现

**音乐来源模式**：
```
[本地文件夹] ←双向同步→ [OSS]
       ↓
   App扫描本地 → 播放音乐
```

**执行**：更新 PRD 和产品路线图，删除"推荐/排行榜/歌单广场"等在线功能，改为"本地+OSS双向同步"模式

### 对话 12：播放统计功能

**Luke**：每个歌曲都要去维护这个嗯，播放次数什么时间播放的，什么时候心情播放，然后播放时长都听了多少次，时间，地点，人物有，啊，当然不用统计那么详细，只要有一个汇总就可以了

**需求要点**：
- 每个歌曲需要维护播放统计（汇总级别，不需要详细到"时间地点人物"那么细）
- 需要记录：
  - 播放次数（听过多少次）
  - 最后播放时间（什么时候听的）
  - 总播放时长（累计听了多久）
- 汇总即可，不需要详细到每次播放的时间/地点/心情

**执行**：更新数据库 schema，新增播放统计表；更新 PRD 功能清单

### 对话 13：播放统计维度细化

**Luke**：就比如说主要的播放时段，喜欢听这首歌的哪一段？然后这首歌是什么分类？啊，就是两个维度，按这首歌的分类去统计，按这首歌本身去统计

**需求要点**：
- **主要播放时段**：Morning (6-12) / Afternoon (12-18) / Evening (18-24) / Night (0-6)
- **喜欢听的段落**：记录哪个时间段被重复播放最多（精确定位到30秒片段）
- **歌曲分类**：用户自定义分类（流行/摇滚/古典/电子/民谣/说唱等），按分类统计

**两个统计维度**：
1. **按歌曲分类统计**：某类音乐在哪个时段播放最多、累计播放时长是多少
2. **按单个歌曲统计**：这首歌的主要播放时段、最爱段落

**执行**：更新 song_stats 表结构，新增字段
